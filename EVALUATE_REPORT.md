## Executive Summary

Tradewars is a well-architected application with clean separation of concerns and 80 passing unit tests. Five findings reach **Critical** severity: the lack of price caching is the root cause of the Massive 429 rate-limiting that breaks games; AUTH_SECRET_KEY silently disables all authentication when unset; python-jose has known CVEs and is unmaintained; SSE has no reconnection logic; and `portfolio_value` silently skips missing prices, distorting game outcomes. The biggest quick wins are a TTL price cache (eliminates 429s), replacing python-jose with PyJWT, and making auth refuse to start without a secret key. Since the prior evaluation, `AUTH_SECRET_KEY` has been set in `.env` and auth is now enforced — but the silent-bypass code path remains a deployment risk.

## Domain Detail

### Architecture & Code Quality

**Finding 5: portfolio_value silently skips missing prices (correctness bug)**
- `accounts.py:192` uses `if ticker in prices:` guard, silently omitting held tickers when prices are missing. This understates portfolio value and distorts P&L / game outcomes. The docstring at line 189 says "prices must cover every held ticker" but the code contradicts it. `test_accounts.py:134-138` expects a `KeyError` that never raises — test confirmed failing: `FAILED - DID NOT RAISE`.
- Severity: Critical | Effort: Low | Impact: High
- Fix: Change `if ticker in prices:` to direct `prices[ticker]` access (raise on missing), or use a fallback to last-known price with a warning.

**Finding 16: test_start_twice_returns_409 broken by auto-stop behavior**
- `test_api.py:156-158` expects 409 when starting a second arena, but `app.py:184-186` now auto-ends the running arena before starting a new one (returns 200).
- Severity: High | Effort: Low | Impact: Medium
- Fix: Update test to expect 200, matching the intentional auto-restart design.

**Finding 20: AUTH_SECRET_KEY copied by value in auth_routes**
- `auth_routes.py:18` does `from backend.auth import AUTH_SECRET_KEY` — this copies the string value, not a reference. If `AUTH_SECRET_KEY` is monkeypatched on the module (as tests do), `auth_routes` retains the original. `app.py:41,153` correctly uses `_auth_mod.AUTH_SECRET_KEY` via module reference.
- Severity: Medium | Effort: Low | Impact: Medium
- Fix: Change to `import backend.auth as _auth` and reference `_auth.AUTH_SECRET_KEY`.

**Finding 21: Private _load_users/_save_users imported cross-module**
- `auth_routes.py:23-24` and `admin_routes.py:15` import underscore-prefixed `_load_users`/`_save_users` from `auth.py`, breaking encapsulation. Any refactor of auth internals breaks two router modules.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 22: os.environ["KEY"] crashes with unhelpful KeyError**

### Security & Compliance

**Finding 2: AUTH_SECRET_KEY silently disables all auth when unset**
- `auth.py:38` sets `AUTH_SECRET_KEY = os.environ.get("AUTH_SECRET_KEY", "")`. When empty, `verify_auth()` returns `"dev"` (line 131-132), `verify_admin()` returns `"dev"` (line 152-153), and the middleware skips checks (`app.py:153-154`). No startup warning, no refusal to bind. A misconfigured `.env` exposes the entire application to the public internet.
- Severity: Critical | Effort: Low | Impact: High
- Fix: Refuse to start when `AUTH_SECRET_KEY` is unset unless an explicit `DEV_MODE=true` env var is present.

**Finding 3: python-jose unmaintained with known CVEs**
- `pyproject.toml:19` depends on `python-jose[cryptography]>=3.3.0` (locked at 3.5.0). CVE-2024-33663 and CVE-2024-33664 affect ECDSA/HMAC algorithm confusion. HS256-only usage mitigates the specific vector, but the library receives no patches.
- Severity: Critical | Effort: Low | Impact: High
- Fix: Replace with `PyJWT` — API is nearly identical.

**Finding 6: Full process environment leaked to MCP subprocesses**
- `mcp_servers.py:52` passes `dict(os.environ)` to Massive MCP. `mcp_servers.py:66` passes `{**os.environ, ...}` to Memory MCP. Both receive `AUTH_SECRET_KEY`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `DEEPSEEK_API_KEY`, `OPENROUTER_API_KEY`, `GOOGLE_API_KEY`, and every other env var. A compromised MCP tool can exfiltrate all credentials.
- Severity: Critical | Effort: Medium | Impact: High
- Fix: Whitelist only required env vars: `MASSIVE_API_KEY` for Massive; `PATH`/`NODE_PATH`/`MEMORY_FILE_PATH` for Memory.

**Finding 7: Docker container runs as root**
- Dockerfile has no `USER` directive. Uvicorn runs as root, expanding blast radius of any code execution vulnerability.
- Severity: High | Effort: Low | Impact: High
- Fix: `RUN adduser --disabled-password appuser` + `USER appuser`.

**Finding 12: JWT token exposed in URL query parameter for SSE**
- `api.ts:111-113` passes the full 7-day JWT as `?token=...`. Token appears in server logs, browser history, Cloudflare edge logs, and Referer headers.
- Severity: High | Effort: Medium | Impact: High
- Fix: Short-lived single-use SSE ticket exchange endpoint, or `fetch`-based SSE polyfill with header support.

**Finding 13: No account lockout after failed login attempts**
- slowapi limits to 5/minute per IP, but no per-account lockout. Distributed brute-force bypasses IP limits. No failed-attempt counter, no temporary lockout.
- Severity: High | Effort: Medium | Impact: High

**Finding 14: Admin delete accepts arbitrary path-style usernames**
- `admin_routes.py:61` uses `{username:path}` — accepts slashes and arbitrary characters. No username format validation on creation either (`admin_routes.py:20-23`).
- Severity: High | Effort: Low | Impact: Medium
- Fix: Add `pattern=r"^[a-zA-Z0-9_.@+-]+$"` to username Field; change `{username:path}` to `{username}`.

**Finding 15: No failed login logging or auth audit trail**
- `authenticate_user` returns `None` silently on failure. No logging on failed or successful logins. No audit trail for admin operations.
- Severity: Medium | Effort: Low | Impact: High

**Finding 30: No security headers**
- No `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, `Referrer-Policy`, or `Permissions-Policy` headers set.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 31: 7-day JWT expiry with no refresh mechanism**
- `TOKEN_EXPIRE_DAYS = 7` at `auth.py:40`. No refresh token, no rotation. Stolen token from logs or XSS gives week-long access.
- Severity: Medium | Effort: Medium | Impact: High

**Finding 32: No financial transaction audit trail**
- `trades` table records trader_id, ticker, quantity, price, ts — but not which human user initiated the arena session.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 33: Username normalization inconsistency**
- `authenticate_user()` at `auth.py:75` normalizes with `.strip().lower()`. `create_user` in `admin_routes.py:48` stores as-is. Case variants create phantom accounts or login misses.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 34: jose/bcrypt silent import fallback**
- `auth.py:21-32` catches ImportError for jose and bcrypt, setting `jwt = None` / `_BCRYPT_AVAILABLE = False`. Combined with `AUTH_SECRET_KEY` bypass, a broken pip install silently disables auth with no startup error.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 35: User credentials in unencrypted JSON with no file locking**
- `users.json` has 664 permissions. `_load_users()` and `_save_users()` do read-modify-write without `fcntl.flock`. Concurrent admin operations can cause data loss.
- Severity: High | Effort: Medium | Impact: Medium

**Finding 47: Supply chain risks in Docker build and MCP**
- `Dockerfile:22` uses `curl | bash` for NodeSource. `Dockerfile:33` installs `mcp_massive` from a git tag (mutable, not SHA). `mcp_servers.py:65` uses `npx -y` which auto-installs latest version at runtime.
- Severity: Low | Effort: Low | Impact: Medium

### Performance & Scalability

**Finding 1: No price caching — every tick and get_state hits external API**
- `prices.py:42-62` has zero caching. The 1 Hz tick loop calls `aget_prices()` for all held tickers across all 4 traders (12-20+ API calls/second). Each trader's `get_state` adds more calls. This is the direct root cause of Massive 429 rate limiting.
- Severity: Critical | Effort: Low | Impact: High
- Fix: Add a 1-2 second TTL cache on `get_price()`. Eliminates >90% of external calls.

**Finding 8: _load_users() reads JSON on every auth check**
- `auth.py:45-53` parses `users.json` on every call. `verify_auth()` triggers 2 reads (decode_token + user check). `verify_admin()` triggers 3 reads. At 1 Hz tick rate, that's 8+ JSON parses/second minimum.
- Severity: High | Effort: Low | Impact: High
- Fix: Cache with file mtime check or short TTL.

**Finding 9: Log view full DOM rebuild on every SSE event**
- `log.ts:27-39` sets `innerHTML = ""` and recreates up to 100 DOM elements per event. With 4 traders active, SSE events arrive several times per second. The forced `scrollTop = scrollHeight` reflow at line 38 compounds the cost.
- Severity: High | Effort: Low | Impact: High
- Fix: Append-only rendering; only clear+rebuild when the ring buffer wraps.

**Finding 26: panel.update() redraws everything on every SSE event**
- `panel.ts:64-81` unconditionally updates portfolio value, heatmap, chart, log, and cycle stats on every SSE event. Most events (tool_called, tool_output, message) don't change portfolio data.
- Severity: Medium | Effort: Low | Impact: Medium
- Fix: Split into `updateLog()` and `updateData()`. Only call `updateData` from tick snapshots.

**Finding 27: No trades table index on trader_id**
- `accounts.py:32-39` creates trades table with only `id` as primary key. `trade_count()` and `trades()` do full scans filtered by `trader_id`. Over a 4-hour game, thousands of rows accumulate.
- Severity: Medium | Effort: Low | Impact: Medium
- Fix: `CREATE INDEX IF NOT EXISTS idx_trades_trader ON trades(trader_id)`.

**Finding 28: SQLite not using WAL mode**
- `accounts.py:59-66` opens SQLite with `check_same_thread=False` but default journal mode. Concurrent traders serialize on write lock. `execute_trade` is synchronous and briefly blocks the event loop.
- Severity: Medium | Effort: Low | Impact: Medium
- Fix: `self.conn.execute("PRAGMA journal_mode=WAL")` after connect.

**Finding 29: MCP subprocess no health monitoring or recovery**
- `trader.py:139-155` holds MCP servers open via `async with` for the entire game (up to 4 hours). If a subprocess crashes, the trader silently dies with no restart.
- Severity: High | Effort: High | Impact: High

**Finding 43: Synchronous HTTP wrapped in asyncio.to_thread**
- `prices.py:54-59` spawns OS threads for each price lookup via `asyncio.to_thread`. With 12-20 concurrent lookups, the default thread pool (40 threads) could saturate.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 53: Single-process scalability ceiling**
- `ArenaHolder` holds one arena instance. SQLite is file-locked to one process. No horizontal scaling path. Appropriate for current use case (1 game, 1-2 viewers) but structural limit.
- Severity: Low | Effort: High | Impact: Medium
