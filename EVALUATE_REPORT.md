## Domain Detail

### Architecture & Code Quality

**Finding 1: `save_users()` truncates file before acquiring lock — race window**
- `backend/auth.py:72-75`: `open(USERS_FILE, "w")` truncates immediately, before `fcntl.flock(f, LOCK_EX)` acquires the exclusive lock. A concurrent reader could see an empty file between truncate and lock grant. The `load_users()` path uses no `LOCK_SH`, so reads are unprotected from write-in-progress.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 2: Login lockout state has no thread lock**
- `backend/auth.py:241-263`. `_login_failures` and `_lockout_until` are plain dicts with no mutex. `_sse_tickets` correctly uses `_ticket_lock` (threading.Lock). Under concurrent login attempts, the failure lists can be mutated from multiple threads simultaneously.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 3: No `trader.py` unit tests**
- No `test_trader.py` exists. `_format_output`, `_extract_usage`, and `_forward_sdk_event` in `backend/traders/trader.py` lines 39–172 have zero direct test coverage. The `neutralize_trader_loop` fixture replaces `run_until_stopped` with a no-op, so the cycle-execution path is untested in the unit suite.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 4: `_format_output()` untested**
- `backend/traders/trader.py:39-57`. The function normalizes two distinct MCP/function-tool output shapes for SSE rendering. It is the critical path for the log display. A SDK-side format change would silently corrupt the SSE log with no test catching it.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 5: `manage_users.py` and admin API cannot set `is_admin` flag**
- `manage_users.py:35-38` hardcodes `is_admin: False`. `admin_routes.py POST /api/admin/users` also hardcodes `is_admin: False`. Admin accounts can only be created by editing `users.json` directly — an operational hazard.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 6: LiteLLM monkey-patch has no automated removal signal**
- `backend/traders/models.py:41`: comment says "Drop when LiteLLM 1.84+ ships" but no test asserts the installed version or fails if the patch becomes unnecessary. If LiteLLM removes `_is_opus_4_6_model`, the patch silently adds a new attribute.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 12: `StopIteration` as failure signal in config resolution**
- `backend/api/app.py:274-275` catches `StopIteration` from `next(...)` in `arena.py:79`. In Python 3.7+, `StopIteration` has special semantics in generators and is converted to `RuntimeError` when propagated through `async for`. Should use `next(..., None)` with explicit check.
- Severity: Medium | Effort: Low | Impact: Low

**Finding 13: No CI pipeline**
- No `.github/workflows/`, no `Makefile`, no pre-commit hooks. 101 tests run in ~10s — fast enough for CI. Without automation, regressions are only caught on manual runs.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 20: pytest in production dependencies**
- `pyproject.toml:15-16`: `pytest` and `pytest-asyncio` listed under `[project].dependencies` instead of `[dependency-groups].dev`. Installed in production Docker image unnecessarily.
- Severity: Low | Effort: Low | Impact: Low

**Finding 21: `app.py` accesses private Arena attributes**
- `backend/api/app.py:252,259`: `arena._started_at` and `arena._final_snapshot` accessed directly. Creates coupling to the Arena internal state machine.
- Severity: Low | Effort: Low | Impact: Low

**Finding 22-25: Minor code quality items**
- `Optional[T]` vs `T | None` inconsistency in `auth.py`. Missing `Path` import in `mcp_servers.py` (masked by `__future__.annotations`). No linting/type-checking config. Mixed import style in `auth_routes.py`.
- Severity: Low | Effort: Low | Impact: Low

**Finding 7: Token usage not persisted to game history DB**
- `backend/environment/accounts.py` `games` table has no token columns. `cycle_end` events carry usage data but it is never summed and stored. Token totals are tracked client-side only and lost on page reload.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 8: History modal shows no token data per game**
- `frontend/src/main.ts:503-541` `renderHistoryGames()` renders date, duration, initiator, and P&L only. No token or cost column. Users cannot see how much a past game cost.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 9: Auth expiry tears down UI without warning**
- `frontend/src/apiClient.ts:52-57`: On 401 after failed refresh, tokens are removed and `auth-expired` event fires. `auth.ts:111-114` calls `_onAuthChange(null)` which redirects to login with no toast or explanation. A running game disappears from view.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 14: Mid-game reconnect loses cumulative token counts**
- `backend/arena/arena.py` `_snapshot()` has no usage field. `ArenaSnapshot` carries no token data. A fresh browser tab opening mid-game starts token counts at zero.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 15: No explicit `game_over` SSE event**
- `backend/arena/arena.py:237` puts `None` sentinel in queue to terminate streams but emits no `game_over` event with final rankings. Clients disconnected near game-end may miss the leaderboard trigger.
- Severity: Low | Effort: Low | Impact: Medium

**Finding 16: Cycle timing not configurable**
- `backend/traders/trader.py:32-34`: `MAX_TURNS_PER_CYCLE = 200`, `INTER_CYCLE_SLEEP_SECONDS = 10.0` hardcoded. No config.json or UI controls to tune these.

### Security & Compliance

**Finding 10: `users.json` world-readable file permissions**
- File is mode 664. `save_users()` creates with default umask. Any local OS user can read bcrypt hashes, email addresses, and admin flags.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 11: Any authenticated user can start/stop the arena**
- `POST /arena/start` and `POST /arena/stop` are protected by JWT middleware only (any valid token passes), not by `verify_admin` or a role check. A non-admin user can abort an in-progress game.
- Severity: Medium | Effort: Low | Impact: Medium

**Finding 17: No `Content-Security-Policy` header**
- `backend/api/app.py:228-234` sets four security headers but omits CSP. Without CSP, any XSS would have no browser-enforced containment.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 38: X-Request-ID echoes client value without sanitization**
- `backend/api/app.py:221`: client-supplied `X-Request-ID` inserted into JSON log output at line 128 without length-capping or character validation. Potential log injection vector.
- Severity: Low | Effort: Low | Impact: Low

**Other Low findings**: HSTS over HTTP unconditionally, auth check after format validation in DELETE, SSE tickets not shared across workers (latent multi-worker risk), no `Permissions-Policy` header, IP-only rate limiting supplemented by per-username lockout.

### Performance & Scalability

**Finding 18: Per-tick N+1 query pattern**
- `backend/arena/arena.py:249-279` calls 6-7 separate SQLite SELECTs per trader per tick (holdings, portfolio_value, cash, pnl, trade_count). With 4 traders, ~28-32 queries per tick. WAL mode prevents blocking, but a batched query would be more efficient.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 19: Single-consumer SSE queue**
- `backend/arena/arena.py:153,339-345`. `Arena.events` is a single `asyncio.Queue`. Two browser tabs calling `GET /arena/stream` race on `events.get()` — each event delivered to exactly one consumer.
- Severity: Medium | Effort: Medium | Impact: Medium

**Finding 26-31: Minor performance items**
- `Prices._cache` no concurrency protection under `asyncio.gather`. `get_config` reads disk every request. `load_users()` called 2-3 times per request chain. Forced reflow in heatmap `flash()`. `getComputedStyle` in uPlot stroke callback. `portfolio_value()` redundant holdings query.
- Severity: Low | Effort: Low | Impact: Low

**Finding 32: SQLite `execute_trade()` blocks event loop**
- `backend/environment/accounts.py:133-191`, `backend/traders/tools.py:79`. Called directly from async function with no `asyncio.to_thread`. WAL reduces contention but write serialization still holds the event loop thread.
- Severity: Medium | Effort: Medium | Impact: Low

**Bottleneck and scalability ceiling**: Primary bottleneck is the Massive REST API (synchronous SDK in thread pool). The 2-second cache TTL limits exposure. Scalability ceiling is the single-consumer SSE queue — the second browser tab gets a degraded stream. Everything else is well-suited to the 4-trader single-arena design.
